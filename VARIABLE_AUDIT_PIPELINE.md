# VeSMed 変数審査パイプライン (Variable Audit Pipeline)

## 概要

全変数を5段階で系統的に審査する。各段階は独立して実行可能。
上流の段階で見つかった問題は即修正し、下流に持ち越さない。

---

## Phase 1: 構造的整合性 (Structural Integrity)

**目的:** 変数定義レベルのバグを排除する

### チェック項目

| # | チェック | 違反例 | 修正方法 |
|---|---------|--------|---------|
| 1.1 | ID重複がないか | S131が2つ存在 | ID振り直し |
| 1.2 | 主変数は全てbinary(absent/present or normal/abnormal)か | S21: absent/burning/sharp/... | binary化 + satellite分離 |
| 1.3 | satellite変数にabsent/not_applicableが含まれていないか | S61: not_applicable/colicky/... | 削除（選ばない=存在しない） |
| 1.4 | 検査変数(L)にnot_doneがあるか（あるべき） | L73: negative/positive のみ | not_done追加 |
| 1.5 | statesの合計確率が1になる設計か | — | noisy-OR構造で自動保証 |

### 判定基準: 主変数のbinary例外

以下はbinaryでなくてもOK（常に判別可能なため）:

- **Severity scale**: mild/moderate/severe（程度は必ず分かる）
- **Laterality**: unilateral/bilateral（どちら側かは必ず分かる）
- **検査結果**: 検査したら結果は確定する
- **Vital signs**: 測定したら値は確定する

### 自動チェックスクリプト
```bash
python audit_variables.py --phase 1
```

## Phase 2: 原子性 (Atomicity)

**目的:** 1変数 = 1臨床概念を保証する

### チェック項目

| # | チェック | 違反例 | 修正方法 |
|---|---------|--------|---------|
| 2.1 | 変数名に「/」「・」「and」が含まれていないか | 悪心・嘔吐 → 悪心 + 嘔吐 | 分離 |
| 2.2 | 変数名に「（原因名）」が含まれていないか | 排尿障害(神経因性) → 尿閉 | 原因を除去、症状名に |
| 2.3 | 1変数のstatesが複数の独立次元を混合していないか | 幻覚: visual/auditory = modality次元のみ → OK | — |
| 2.4 | absent/presentの対概念が本当に1つか | 幻覚/妄想 = 感知障害/思維障害 | 分離 |

### 判定フロー

```
変数名に「/」「・」がある
  → 左右は臨床的に独立して出現しうるか？
    → YES → 分離（例: 悪心・嘔吐 → 悪心 + 嘔吐）
    → NO → 統合OK（例: 肋骨脊柱角叩打痛 = 1つの手技）
```

```
変数名に「（補足）」がある
  → 補足は原因/機序を述べているか？
    → YES → 原因を除去し症状名のみに（例: 排尿障害(神経因性) → 尿閉）
    → NO → OK（例: 出血傾向（紫斑/歯肉出血/鼻出血） = 例示、分離検討は別途）
```

## Phase 3: OPQRST完全性 (OPQRST Completeness)

**目的:** 主要症状の全鑑別次元がsatelliteで表現されているか確認する

### 対象: edge数15以上の主症状

各主症状について以下の6次元を確認:

| 次元 | 略 | 問い | 例 |
|------|---|------|---|
| Onset | O | 発症様式は？ | T02で全局的にカバー。個別追加は稀 |
| Provocation/Palliation | P | 何で悪化/改善？ | 食後増悪, 体動増悪, 前傾改善 |
| Quality | Q | どんな痛み/感覚？ | colicky, burning, sharp, dull |
| Region/Radiation | R | どこ？放散は？ | epigastric, 背部放散, 右肩放散 |
| Severity | S | どのくらい？ | mild/moderate/severe |
| Timing | T | 時間経過は？ | constant, intermittent, progressive |

### チェックマトリクス

各主症状 × 6次元で、satellite変数の有無を○×で記録:

```
           O   P   Q   R   S   T
S05 頭痛   T02 S68 S60 -   S05 S69  → R欠落？(放散は頭痛では不要 → N/A)
S12 腹痛   T02 S62 S61 S89 S64 S65  → 全完備 ✓
S21 胸痛   T02 S50 S83 S51 -   -    → S,T欠落 → 追加検討
...
```

### N/A判定基準

その次元がその症状の鑑別に臨床的に寄与しない場合はN/Aとする。
例: 頭痛のRadiation（放散は鑑別にほぼ寄与しない）

## Phase 5: 疾患逆引き検証 (Disease-Driven Validation)

**目的:** 各疾患の鑑別に必要な変数が全て存在するか確認する

### 方法

345疾患を1つずつ、以下を確認:

1. その疾患のkey distinguishing feature（教科書的な特徴的所見）を列挙
2. 各featureに対応する変数IDを確認
3. 対応する変数がなければ → **変数追加が必要**
4. 対応する変数はあるがedge/CPTがなければ → **edge/CPT追加が必要**

### 優先度

| 優先度 | 基準 |
|--------|------|
| HIGH | 疾患のpathognomonic sign/症状に対応する変数がない |
| MED | 疾患の common feature に対応する変数がない |
| LOW | 疾患のminor feature に対応する変数がない |

### 実施方法

全345疾患を手動で確認するのは非現実的。以下で効率化:

1. **Edge数が少ない疾患（<10 edges）から着手** — 変数不足が最も深刻
2. **FATAL/CntMis案例の正解疾患から着手** — 誤診の直接原因
3. **疾患カテゴリ単位でバッチ処理** — 同カテゴリは共有所見が多い

```bash
# Edge数少ない疾患を抽出
python audit_variables.py --phase 5 --low-edge-diseases
```

## 実行順序

```
Phase 1 (構造) → 即実行可能。自動スクリプトで検出・修正
    ↓
Phase 2 (原子性) → 変数名を目視レビュー。「/」「・」検索で効率化
    ↓
Phase 3 (OPQRST) → edge数15+の主症状のみ対象。マトリクス埋め
    ↓
Phase 4 (系統別) → ROS標準リストとの突合。最も網羅的
    ↓
Phase 5 (疾患逆引き) → edge不足疾患から。最も労力大だが最も効果大
```

Phase 1-3は変数層、Phase 4は変数+edge層、Phase 5はedge+CPT層に主に効く。

---

## 鉄律 (Iron Rules)

1. **1変数 = 1臨床概念** — 「A/B」「A（原因）」は許さない
2. **主変数は binary** — absent/presentまたはnormal/abnormal。描述的stateはsatelliteへ
3. **satelliteにabsentなし** — 主変数がabsentならsatelliteは入力しない
4. **検査変数にはnot_doneを設ける** — 「検査してない」と「検査して正常」は別情報
5. **変数名は症状/所見で記述** — 原因・機序・診断名を含めない
6. **重複計算を防ぐ** — 主変数=absentのとき、そのsatellite群は一切入力しない
