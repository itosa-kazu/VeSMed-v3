# 新変量追加ワークフロー

$ARGUMENTS に変量名（日本語）とstates案が渡される。

## 原則: 1変量ずつ、全疾患照合必須

**変量追加は疾患追加より影響が大きい。** 1変量が全376疾患のスコアに波及する。
疾患追加=1疾患→15辺。変量追加=1変量→50+辺。慎重に。

## 完全サイクル（省略禁止）

### Phase 0: 変量仕様定義（ユーザ承認必須）

1. **ID割当**: 次の空きIDを確認（L105, S170, E98等）
2. **State設計**: 文献ベースで臨床的に意味のある分類を設計
   - 検査値変量に `not_done` ステートは禁止
   - `negative`（陰性）は有効な臨床結果であり `not_done` とは異なる
   - State数は3-7が最適。多すぎるとCPT精度が落ちる
3. **Leak分布設計**: 「どの疾患も原因でない場合」の背景分布
   - 文献の一般人口有病率から設定
4. **臨床的意義の確認**: この変量が鑑別にどう貢献するかをWebSearchで調査
   - 感度/特異度のデータを収集
   - どの疾患群を分離できるかを明確化
5. **ユーザに提示して承認を得る**（変量定義は一度commitしたら変更コストが高い）

### Phase 1: 全疾患スクリーニング（**最重要・省略禁止**）

**全疾患をcategory_sub別にグループ化し、系統的にスクリーニングする。**

```python
# カテゴリ別グループ化の例
groups = {}
for disease in all_diseases:
    cat = disease.get('category_sub', 'other')
    groups.setdefault(cat, []).append(disease)
```

各グループについて：
1. **関連性判定**: このグループの疾患がこの変量の値を**直接引き起こすか？**
   - YES → Phase 2でCPT設計
   - NO → スキップ（理由を記録）
   - MAYBE → 文献確認
2. **間接因果は辺にしない**: 転移/DIC/ショック経由の間接効果は直接辺にしない
3. **判定結果を一覧表にまとめる**:

```
| category_sub | 疾患数 | 関連 | 辺追加対象 | 理由 |
|-------------|--------|------|-----------|------|
| bacterial_infection | 45 | YES | 30 | 細菌感染は鏡検陽性 |
| viral_infection | 20 | NO | 0 | ウイルスは鏡検陰性 |
| autoimmune | 15 | NO | 0 | 非感染性 |
| fungal_infection | 8 | YES | 8 | 真菌は鏡検で検出可能 |
```

### Phase 2: バッチCPT設計（文献ベース）

**同じ鏡検結果を示す疾患群をバッチで処理。**

1. **菌形態グループ別に文献検索**:
   - GPC群: 肺炎球菌/黄色ブドウ球菌/連鎖球菌感染 → Gram染色感度を一括調査
   - GNR群: 大腸菌/クレブシエラ/緑膿菌 → 同上
   - AFB群: TB/NTM → 抗酸菌染色感度
   - 分岐フィラメント群: ノカルジア/放線菌
   - 真菌群: アスペルギルス/カンジダ/クリプトコッカス

2. **CPT値は文献の感度データから直接設定**:
   - 良い例: `"肺炎球菌肺炎: 喀痰Gram GPC 30-50%(PMC7384319)"`
   - 悪い例: `"肺炎球菌肺炎: GPC"`（根拠なし）

3. **バッチ内の整合性確認**: 同じ菌形態グループ内でCPT値が矛盾しないか

### Phase 3: 実装（step1/step2/step3同時更新）

1. **step1**: 変量定義追加
2. **step2**: 全辺追加（Pythonスクリプトでバッチ処理推奨）
3. **step3**: leak + 全parent_effects追加
4. **validate_edges.py**: EDGE_NO_CPT = 0, CPT_NO_EDGE = 0を確認
5. **IDF健全性**: parent数 ≥ 3（理想は5以上）
   - parent不足なら臨床的に妥当な辺を追加

### Phase 4: 既存案例レトロフィット

**既存案例のvignetteをスキャンし、新変量の情報があれば追加。**

1. `real_case_test_suite.json`の全案例をキーワードスキャン
   - 例: sputum_microscopy追加時 → "Gram", "AFB", "抗酸菌", "塗抹" 等を検索
2. **辺の存在確認**: evidence追加前に、期待疾患がその変量への辺を持つか確認
3. **辺がなければ先に辺追加 → それからevidence追加**（feedback_evidence_needs_edges原則）
4. **1案例ずつ慎重に**（一括自動追加禁止）

### Phase 5: 回帰テスト + WebUI

1. `python bn_inference.py` で全案例テスト
2. MISSリストを前回と比較。新たなMISSは原因分析
3. Top-1/Top-3が2%以上低下したら辺監査
4. **WebUI STATE_JA**: 全stateの日本語ラベルをapp.pyに登録
5. `python validate_cases.py` でERROR 0確認
6. FATAL 0確認

### Phase 6: commit push

commitメッセージに: 変量名、辺数、parent数、影響案例数、Top-1、Top-3、FATAL

## IDF健全性ルール

| parent数 | IDF | リスク | 対応 |
|---------|-----|-------|------|
| 1-2 | ≈1.0 | 特定疾患を偏袒 | **追加辺必須** |
| 3-5 | 0.6-0.8 | やや高いが許容 | 監視 |
| 6+ | <0.6 | 健全 | OK |

## 変量設計の原則

1. **State数は3-7**: 少なすぎると情報量不足、多すぎるとCPT精度低下
2. **互斥・網羅**: 全stateが相互排他的で、観察可能な結果を網羅
3. **臨床的意味のある閾値**: 検査値のカットオフは臨床ガイドラインに従う
4. **確定診断検査は変量にしない**: 活検結果、特異的PCR等は除外
5. **予備的検査はOK**: Gram染色、画像所見、迅速検査等は変量にできる
6. **原子化原則（必須）**: 1変量=1つの臨床観察単位。異なる検体・手技・感度を持つ検査は別変量にする
   - 悪い例: `sputum_BAL_microscopy`（喀痰とBALは検体品質・感度・臨床意義が全く異なる）
   - 良い例: `sputum_gram_stain` + `BAL_microscopy`（別変量）
   - 判断基準: 同じ結果(例:GPC陽性)でも**診断的意義が異なる**なら別変量
   - 例外: CXR(PA/側面/portable)は同一フィルムの読影方法差のみ→統一OK

## 注意事項

- **1変量ずつ追加**: 複数変量を同時に追加すると回帰分析が困難
- **段階的拡張**: まず最重要な疾患群(10-20)に辺を張り、回帰テスト。問題なければ残りを追加
- **過拟合禁止**: テスト案例のためだけの辺は追加不可。臨床的合理性が独立に必要
- **revert禁止**: 回帰悪化してもユーザ承認なしにrevertしない
