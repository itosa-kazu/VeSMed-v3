## 鉄則：大規模作業は並列Agent分割

### 適用範囲

- CPT手動設計（1Agent=1変量）
- 辺監査（1Agent=1疾患群）
- 案例検索（1Agent=1疾患）
- 疾患追加（1Agent=1疾患）
- States再設計
- その他、独立した単位に分割可能な作業全般

### やり方

1. 作業を独立した単位に分割（変量ごと、疾患ごと等）
2. 各単位に1つのAgentを起動（最大5並列）
3. 各Agentに現在の状態と臨床コンテキストを渡す
4. Agentが臨床根拠に基づいて結果をJSON形式で返す
5. 結果を収集して一括適用 → 回帰テスト

### Agentへの指示テンプレート（共通原則）

- 対象の現在の状態を完全に提示（CPTなら全parent_effects、辺なら全既存辺等）
- 臨床的コンテキストを明記（変量の意味、疾患の特徴等）
- 「機械的変換は禁止、臨床的に考えろ」と明記
- 出力形式はJSON only（markdownや説明不要）
- ファイルを直接編集せず、JSONを返すだけにする

### バッチサイズとテスト

- 5単位ずつバッチ処理
- 各バッチ後に `python bn_inference.py` で全案例回帰テスト
- 退化があれば該当単位を個別調整
- 全バッチ完了後に最終回帰テスト

### 工数が大きい単位の扱い

- parent_effectsが多い変量（例: L04胸部X線の79親）は1Agent丸ごと担当
- 変量数が多い場合（例: 71変量のnot_done清理）は5変量ずつバッチ処理

**Why:** 76変量のnot_done一括削除で機械的変換を3回試みて全て退化。1変量ずつ手動設計が正解だが、逐次では遅すぎる。並列Agentなら5単位を同時に臨床設計できる。CPTに限らず、辺監査や案例追加など独立分割可能な大量作業すべてに適用可能。
**How to apply:** 独立分割可能な大量作業が発生したら、Agent toolで並列起動。各Agentに1単位を担当させる。
