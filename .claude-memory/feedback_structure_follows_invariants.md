## 三鉄律（通用、全プロジェクト適用）

### 鉄律1: 結構は領域不変量(Domain Invariants)のみに依拠する

構造的決定（テーブル構造、インターフェース定義、state設計）は領域の**公認の不変事実**のみに基づく。
当面の便宜・データ可得性・実装の簡易性には基づかない。

判定基準: **「10年後もこれは成立するか？」** Yes → 構造に。不確か → データに。

VeSMed適用: 変量statesは医学標準cutoff（WBC>10000=白血球増多）に基づく → 10年後も成立 → 構造にして良い。

### 鉄律2: 演進は常に加法(Additive-Only)

- ✓ 新変量追加、新辺追加、CPT値更新、新疾患追加
- ✗ 既存変量のstate変更、インターフェース契約変更、推論コア公式変更

「禁止の演進」が必要になったら → 鉄律1違反のサイン → 構造を局所的に再設計（補丁ではなく）

### 鉄律3: 補丁 = 架構bugの症状

- 補丁が必要 = 関節の切り方が間違っている
- 対処: 補丁を重ねるのではなく、その関節を局所的に再設計
- 鉄律1に従えば切り間違いは稀 → 局所再設計も稀

## 背景原理

**"Carving nature at its joints"（Plato）**

システムの構造的境界は、ドメインの自然な境界と一致すべき。
自然な関節で切れば、各関節内の進化は局所的。
恣意的な線で切れば、進化が横断的関心事を生み、構造的書き直しを強制する。

**Why:** VeSMed大清洗の教訓。901変量のstate設計が「便宜」に基づいていたため、
辺+CPTの全面やり直しが必要になった。医学標準に基づくstate設計なら構造は安定する。

**How to apply:** 新しい構造的決定をする前に必ず問う:
「これはドメイン不変量か、それとも現在の便宜か？」
不変量 → 構造に焼き込んでよい。便宜 → データ/設定として外出しする。
